VerifyFlow · 配套提词稿
讲解者专用 · 对应 verifyflow-explainer.html · 逐屏怎么讲 / 被问怎么答
怎么用这份稿子：把 verifyflow-explainer.html 投屏给观众，这份提词稿放你手边或另一块屏。 每张卡对应演示页的一屏，颜色边表示听众侧重 — 蓝＝技术细节 · 黄＝价值/管理视角 · 灰＝通用/过场。 时间紧就只讲蓝+黄的核心屏；🗣 是可以照着说的口播。全程纪律见最底部，是这次讲解的灵魂。

开场 30 秒：电梯陈述（背下来）

"代码审查回答『写得对不对』，VerifyFlow 回答『工单要求的事，到底做到了没』—— 它把 Linear 工单的验收标准逐条拎出来，在真实 checkout 上实际执行验证、留下证据，给出一个能进合并门的交付判决。"

看人调主线

👩‍💻 工程师"真执行、不读 diff 猜"+三个验证级别+证据可追溯。重点屏：05、06、09、10。
👔 管理 / Lead替掉"人肉核对工单"、能进合并门做无人值守闭环、诚实的收益边界。重点屏：04、07、10、13。
🧪 想试用的人装上拿到 vf、四条命令起步、auto 选级别。重点屏：08、09，结尾给 npx 一行。
屏 01开场 / Hero — 用证据说话 过场~30s#top
目的：一句话定位，让全场知道接下来 10 分钟在解决哪个具体问题。
🗣 口播
  • 念电梯陈述那句，然后停一拍。
  • "今天不讲它怎么实现，讲三件事：是什么、有没有用、怎么用。"
钩子：先抛痛点——"PR 看着没问题就合了，但工单里要求的那几条，真没人逐条核对过。" 大多数人会点头。
屏 02这是什么 — 一句话讲清楚 通用~1min#what
目的：把"它不是什么"说清，避免被误当成测试工具或代码审查。
🗣 口播
  • "它不再跑一遍测试，不看代码风格。"
  • "它做四件事：抽验收标准 → 真 checkout 实际执行 → 留证据 → 出交付判决，而且积累记忆，越用越准。"
一句记忆点："代码审查说『写得对』，VerifyFlow 说『做到了没』。"
屏 03目录 — 这页讲什么 过场~20s#toc
目的：给地图，别逐条念。手一划带过。
🗣 口播
  • "分这么几块——定位、级别、效率、怎么装怎么用、真实案例、横向对比、边界。挑你们最关心的看。"
屏 04三件套定位 — 各管一段，不重叠 价值~1.5min#fit
目的：划清 Symphony / Crosscheck / VerifyFlow 的边界——这是最容易被混淆、也最显价值的一屏。
🗣 口播
  • "Symphony 派活，Crosscheck 管『代码写得对不对、能不能合』，VerifyFlow 管『工单承诺兑现没』。"
  • "关键区别——代码很漂亮但功能没做全：Crosscheck 可能放行，VerifyFlow 会拦。"
这屏是把 VerifyFlow 和"普通 code review"切开的地方，慢一点讲。
Q：这不就是 code review 多一步？
不同问题。review 看代码质量；VF 看"工单这几条做到没"，靠真执行，不看 diff。
屏 05怎么运转 — 一次验证里的 6 步 技术~1.5min#run
目的：让人信"它真的跑了"，并埋下"保守是设计原则"这条主线。
🗣 口播（顺着 6 个图标走）
  • "①抽 AC → ②checkout PR head 真跑命令/测试/浏览器，不是读 diff 猜 → ③日志/截图/trace 全存档 → ④逐条判 pass/fail/partial/blocked/not_evaluable → ⑤合并门 → ⑥测试点进记忆。"
"保守是设计原则：拿不准就判 blocked / not_evaluable，绝不编一个假失败。证据不够就老实说『没法验，交给人』。" ← 这句反复出现，是它的人格。
Q：我怎么知道它真跑了、不是 LLM 编的？
每条判决都能追到证据：probe 日志里有 $ 命令 / cwd / exit / executed:true + 原始输出；UI 步有截图。没真跑过的会标成 blocked，不会假装通过。
屏 06三个级别 — functional / ui / journey 技术~2min#levels
目的：讲清"同一条标准能验浅也能验深"，以及 auto 会自动选、跑不了就降级并说明。
🗣 口播
  • functional：后端/CLI/库——真 checkout 跑命令探针 + 项目自己的测试，看退出码。最快最稳。
  • ui：Playwright 开真浏览器，AI 驱动点击/填表/读页面；需要跑着的 app（--base-url），登录态用 --ui-auth。
  • journey：跨多步、跨前后端的完整旅程（注册→下单→看到结果）。最慢，最接近真实。
  • auto：读工单挑级别；没有浏览器就降级到 functional，并在报告里说明原因——不假装验过。
举具体例子最好懂："『点导出按钮能下载 CSV』就是 ui；『用户从登录到支付完成』就是 journey。"
Q：随机性问题？（浏览器/AI 驱动）
承认：结果有随机性、已立项在修；journey 里一个 fail 要"多次独立重跑能复现"才算数，正是为了压住抖动。
屏 07提没提效率 — 含真实数据 + 诚实 caveat 价值~2min#eff
目的：讲清它替掉的是"人肉确认工单"，并能拼成无人值守闭环；同时主动亮出数据的局限（这反而加分）。
🗣 口播
  • "没它的时候：reviewer 看完代码还得翻工单逐条核对、手动拉分支跑一遍点一遍——经验还留不下来。"
  • "有它之后：标准自动逐条核对带证据贴回 PR，失败能阻断合并，测试点进记忆下次秒复用。"
  • 闭环一句话：vf watch --auto-merge 盯着 Crosscheck 批过的 PR，验证通过就 squash-merge——从工单到已验证合并，中间可以没有人。
📊 真实指标（可被核对，别夸大）
  • 39.1% 记忆复用率 · 47.8% 标准 pass 率 · 37% not_evaluable（也暴露了保守的短板）
  • 4 次 dogfood · 46 条标准 · verdict：needs_fix×2 / manual_review_required×2
主动认怂这句一定要说，会显著加可信度："还没有『省了多少工时』的规模化实测——这 4 次都是给自己验，样本小。机制跑通了，硬收益要真实团队 repo 跑一段才有。"
屏 08安装 — 拿到 vf 命令 过场~30s#install
目的：试用者才细讲，其他场合一句带过。
🗣 口播
  • "三种：A 零安装 npx 先试一眼；B clone + npm link 拿到全局 vf（推荐）；C npm 全局（还没发布）。"
  • 提一句前置：Node≥20、gh 已登录、读 Linear 要 LINEAR_API_KEY（或 --fixtures 离线）。
方式 B 的坑先提醒："git pull 之后要 npm run build，否则 vf 跑的是旧编译产物。"
屏 09怎么用 — 本次真实跑过的命令 技术~1.5min#use
目的：强调"这些不是示意，是本项目真跑过的命令和输出"。
🗣 口播（四条主线命令）
  • ① vf doctor 先体检环境。
  • ② vf watch --auto-merge 挂着自动盯——念那段真实 tick 输出，"Crosscheck 批了 #66，下一轮它自动接手验证。"
  • ③ vf run --pr …#66 --checkout --comment 验某个 PR、真 checkout、把判定表贴回评论。
AC-1 ✓ pass          0.92  Quickstart leads with vf demo …
AC-5 ✗ fail          0.85  Setup states vf onboard saves the key …
AC-9 ? not_evaluable 0.80  typecheck / test / build pass …
指着 --comment 说："判定表直接贴回 PR，证据留在 GitHub 上，不是聊天框里看一眼就没了。"
屏 10真实案例 — dogfood，好坏都暴露 价值~2min#case
目的：全场最有说服力的一屏——真实发生、不是演示数据。讲完三个小故事。
🗣 口播（A→B→C 三段）
  • A 抓到真问题（--version PR）：Crosscheck 标 NEEDS WORK——缺测试、help 没列 --version、import 风格不一致，全真该补，补完 APPROVE 合并。
  • B 暴露误判（README PR）：验文档型标准时去跑 vf onboard，沙箱把密钥剥了 → exit 1 → 误判成失败，其实 README 是对的。这个误报被真实复现了。
  • C 误判催生新能力：为解决 B，当场设计并合入人类反馈回路 vf feedback——标一次 AC-5，重验后那条从 fail 降为 blocked，整体从 needs_fix 变 manual_review_required，PR 评论自动刷新。判错→一句话纠正→永久生效。
收这屏的金句："它不完美——但它把『判错了』也变成了能被一句话修正、且会被记住的事。"
屏 11总体对比 — vs LLM agent / CI / 人工 通用~1min#vs
目的：别逐格念表。挑 3 个差异化维度 + 1 个诚实短板。
🗣 口播（挑这几行）
  • "绑的是工单验收标准，不是 diff/测试——这是 CI 不知道的事。"
  • "真执行 + 证据 + 记忆，比无状态 LLM judge 扎实；还能接合并门。"
  • 诚实短板："『内部逻辑型』标准它会瞎编 shell 命令，不如直接跑单测准；而且偏保守，一堆 not_evaluable 要人收尾。"
"它是补位，不是顶替 CI 和代码审查。"
屏 12vs 现成 skill — verify / CodeRabbit / Playwright-MCP 技术~1.5min#vs-skills
目的：用两个案例说清差异轴——绑不绑工单 / 执不执行 / 留不留证据记忆 / 接不接合并门；并大方承认它输的地方。
🗣 口播
  • 案例①（功能）："CodeRabbit 读 diff 不执行、不知道工单那 12 条；verify skill 跑完即忘不贴回 PR；VF 自动抽 12 条→逐条跑→贴回→AC-5 误判被 feedback 纠正。"
  • 案例②（UI）："Playwright-MCP 能开浏览器点按钮、而且最强，但它不知道这对应工单哪条、不出判决、不留记忆、不接门；VF 的 ui 级别底层就建在 Playwright 上，加的是『绑工单→判交付→留证据→进门』那一层。"
承认输的地方（务必说）："代码/安全审查深度——它根本不审，CodeRabbit 完胜；速度——它慢（checkout+执行约 1 分钟/次）；成熟度——早期、4 次 dogfood、还没发 npm。"
Q：那我为什么不直接用 Playwright-MCP？
不是竞争关系。要"开浏览器干活"用它；要"这步证明了工单某条标准、且能进合并门"才用 VF——VF 在它上面加判决层。
屏 13边界 + 收尾 — 它不做什么 价值~1min#limit
目的：用边界反向强化定位，干净收尾。
🗣 口播
  • "不审代码（Crosscheck）、不拆工单派 agent（Symphony）、不动钱不发生产邮件不做不可逆副作用、不替代 CI。"
收尾金句："在『Symphony 写 → Crosscheck 审 → VerifyFlow 验』这条链上，它补的是最后也最关键的一环——交付是否为真。"
🎬 留给观众的一句 + 行动
  • "想自己跑一眼，离线零配置：npx github:humanbased-ai/verifyflow demo。"

⚠ 全程纪律（这次讲解的灵魂，别破功）

  • 不夸大收益：没有工时数据就说没有。"机制跑通、规模化待验"是诚实且更可信的说法。
  • 主动认短板：随机性、偏保守(37% not_evaluable)、对内部逻辑型标准会瞎编命令、还没发 npm——被问之前自己先说。
  • "真执行 + 证据"是主心骨：任何"它怎么不被骗"的质疑，都回到 probe 日志(exit/executed:true)和截图。
  • 定位别漂移：它不是 code review、不是 CI、不是测试框架——是"工单兑现没"的交付判决层。
  • 对比要公平：每个对比对象都先肯定它强在哪，再讲 VF 的差异点。不踩别人抬自己。
