## 三、结果：工具判了"不合格、不让合并"，但这是误判

工具最终裁决：**needs_fix / 🔴 不让合并**，理由是有 4 条验收标准没通过。

**我逐条手动复核后发现：这 4 条 PR 其实都做到了，是工具自己判错了。**

| 验收标准 | 工具裁决 | 手动复核真相 |
| --- | --- | --- |
| 缺配置文件时不崩溃 | ❓ 没法判定 | ✅ 实测正常，不崩 |
| 带了单元测试 | ❓ 没法判定 | ✅ 3 个测试全过 |
| 支持自定义路径参数 | ❌ **不合格** | ✅ 实测明确支持 |
| 缺文件时打印"未找到" | ❓ 没法判定 | ✅ 实测正确打印 |

**正确结论应是"合格 ✅"，工具却给了"不合格 🔴"。**

### 为什么判错？

工具**自己生成的测试命令写得不严谨**，结果没测到 PR 里的新代码、而是测到了环境里的旧程序，于是旧程序报的错被当成了"PR 的毛病"。

> 打个比方：它想尝这锅菜好不好吃，结果舀错了锅、尝了隔壁那锅馊的，然后说"这道菜不行"。菜其实是好的。

## 四、这说明什么（关键，也是汇报重点）

这次"翻车"反而是最有价值的发现，正好对上 Karpathy 的论断：

> **模型没错、PR 没错，错在"验证这道关"做得不够严——结果整个裁决就失真了。**

所以有两点可以明确跟老板讲：

1. **架子搭通了，方向对** —— "把真实 PR 拉下来真跑、出证据、给合并裁决"这条主线能转，这是我们和市面上"只读代码、不真跑"的工具的核心差异。

2. **它替我们暴露了自己头号要修的毛病** —— 验证命令必须锁死运行环境，否则会误杀好 PR。这是一个用真实工单跑出来的、可复现的改进点，已对应到我们改进文档里"环境契约"和"失败分诊"两条优先级。比凭空列清单有说服力得多。

## 五、已完成的修复（分支 `fix/probe-env-contract-and-triage`）

发现 → 修 → 验证，闭环已完成。三道防线（对应改进文档 §2.1 环境契约、§3.3 失败分诊）：

1. **命令拼装修正**：复合命令（带 `;`、shell 赋值、管道）整条用 `uv run sh -c '…'` 包起来跑，保证里面的 `symphony`/`python`/`pytest` 全部用项目环境，不再敲到系统旧版。
2. **失败分诊**：探针报的若是"命令/环境错误"（`unrecognized arguments`、`No module named`、`command not found` 等），判为"工具自己的问题、需复核"，**绝不算成 PR 的 fail**。
3. **裁决复核加护栏**：复核环节不允许仅凭一条"报错的探针"就把某条标准打成 fail——只有在"权威探针干净跑完、结果确实相反"时才允许判 fail。

并补了两个回归测试，把这个误判**永久锁死**，不会重犯：
- `src/core/planner/repoConfig.test.ts`（命令拼装）
