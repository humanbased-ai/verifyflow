A · 完整写法 — issue 与解析结果

带 ## Acceptance Criteria 段，命令用反引号包起来

工具直接把每条标准变成一个独立 probe，命令可以照搬运行。

// issue.description ## Summary `greet --version` should print the tool name and version so users know which build they run. ## Acceptance Criteria * `node bin/greet.mjs --version` prints `ExampleCLI <version>` (e.g. `ExampleCLI 1.2.3`) and exits 0. * Version value is read from package.json metadata — not hardcoded.
// vf run --dry-run 输出 ## Resolved acceptance criteria (2) - AC-1 [linear_explicit, backend, observable]: `node bin/greet.mjs --version` prints `ExampleCLI <version>` ... exits 0. probe: node bin/greet.mjs --version (from ticket) - AC-2 [linear_explicit, backend, observable]: Version value is read from package.json metadata — not hardcoded.
结果：2 条标准，AC-1 的 probe 直接是 ticket 里给的命令——验证阶段可以原样执行并对照预期。AC-2（"不许硬编码"）这条没有 ticket 也根本不会被检查到。
B · 只有 Summary — issue 与解析结果

只保留概述，没有显式 AC 段

工具退而求其次，从描述里硬找一句包含命令模样的话当 probe——结果不一定靠谱。

// issue.description ## Summary `greet --version` should print the tool name and version so users know which build they run.
// vf run --dry-run 输出 ## Resolved acceptance criteria (1) - AC-1 [linear_explicit, backend, observable]: `greet --version` should print the tool name ... probe: greet --version (from ticket — but this command is wrong: real PR needs `node bin/greet.mjs --version`)
结果：标准从 2 条砍到 1 条。probe 命令从 Summary 反引号里抠出来的是 greet，但 PR 的实际入口是 node bin/greet.mjs——跑下去会直接失败，不是因为代码错，是因为标准本身错。
C · 完全模糊 — issue 与解析结果

没有任何技术细节

工具走最后兜底：把标题/首句当占位标准，并显式记录"ticket 质量不达标"。

// issue.description Users have been asking for this. Let's add it.
// vf run --dry-run 输出 ## Resolved acceptance criteria (1) - AC-1 [linear_implicit, backend, observable]: Users have been asking for this. Let's add it. ## Ticket quality notes - No explicit acceptance criteria found in the Linear issue; criteria inferred from summary.
结果：来源标签从 linear_explicit 掉到 linear_implicit；没有 probe 命令；质量警告写进报告。即便实际跑代码全过，verdict 也被锁在 manual_review_required，拿不到 accept。

为什么 LLM 不能替你补

VerifyFlow 的 LLM 富化阶段 (src/core/criteria/parser.ts:197) 的 system prompt 写得很死：

"The Linear issue is the ONLY source of truth for acceptance criteria ... Do not invent criteria the issue does not support."

原因很简单：如果允许 AI 一边看着 PR 改了什么、一边自动生成"该验证什么"，那就是后视镜出题——只会出能通过的题。工具的核心价值是"在写代码之前定下的合同"和"写代码之后的产物"之间做对账。

结论

Linear issue 里的验收条件不是文书工作——它是 VerifyFlow 用来验证 PR 的合同。三组实测显示：

  • 没有验收条件，工具就是瞎跑：probe 命令错的、覆盖项目少一半、verdict 被锁顶。
  • AI 不会替你补：被故意限死，防止 AI 自己出题自己答题。
  • 建议写法：在 issue 描述里加一个 ## Acceptance Criteria 段，每条用反引号包"可观测命令 + 预期输出/退出码"。例：运行 `node bin/x.mjs --version` 应输出 `Foo 1.2.3` 退出码 0。
  • 顺序很重要：先写 issue → 人过一眼 → 再写 PR。倒过来等于白做。
