evidence-backed delivery verification

代码审查说"写得对",
VerifyFlow 回答"做到了没"。

它只解决一个问题:

这个 PR,真的把它关联的那张 Linear 工单要求的事做出来了吗?

不是再跑一遍测试,也不是看代码风格。VerifyFlow 把工单里的验收标准逐条拎出来, 在真实 checkout 上实际执行验证、留下证据,最后给出"交付判决"——并随着每次运行积累可复用的记忆,越用越准。

怎么运转

一次验证里发生了什么

📋1. 提取验收标准

从 Linear 工单(或 PR 描述)拆出一条条可验证的标准 AC。

⚙️2. 真执行探针

checkout PR head,跑命令 / 测试 / 浏览器操作,而不是只读 diff 猜。

📎3. 留证据

命令日志、截图、trace 全部存档,每条判决都能追溯到证据。

⚖️4. 逐条判决 + 汇总

pass / fail / partial / blocked / not_evaluable → 汇总成交付结论。

🚦5. 合并门

advisory(只报告)/ merge_gate(失败阻断)/ strict(更严)。

🧠6. 积累记忆

把可复用的"测试点"存下来,下次相同标准直接复用——越用越准。

怎么知道验的是哪张工单?——从 PR 反推。VerifyFlow 读 PR 正文里的 Linear 链接,或分支名里的工单号(如 haol/in-123-…)来定位工单。
方向很关键:一张工单常常对应多个 PR(拆成几次提交)。所以"PR → 工单"是唯一确定的——每个 PR 自带它要兑现的那张工单;反过来"工单 → PR"是一对多,会有歧义。
前提:这个 PR 得能解析出工单。解析不到时:要么 --linear IN-123 手动指定,要么 --allow-no-ticket 退而按 PR 自己的描述验(结论上限 manual_review_required)。
保守是设计原则:拿不准的判 blocked / not_evaluable,而不是编一个假的失败。证据不够,就老实说"没法验,交给人"。
验到多深

三个验证级别:从"命令能跑"到"用户走通"

同一条验收标准,可以验得浅、也可以验得深。你可以指定级别,或交给 auto 看工单内容自动选。

级别验什么怎么验(真执行)需要速度
functional 功能验证
后端 / CLI / 库的行为		 真 checkout PR head,跑命令探针 + 项目自己的测试套件,看退出码/输出 checkout / workdir 快
ui UI 验证
页面在浏览器里真的对		 Playwright 开真浏览器,AI 驱动点击/填表/读页面,判断界面行为是否满足标准 Playwright + 运行中的 app(--base-url);需登录态用 --ui-auth 中
journey 端到端验证
一条完整用户旅程		 多步骤串联:后端操作 + 浏览器操作走完整流程(如 注册→下单→看到结果) checkout + Playwright 慢
auto 自动选级别 读工单标准挑最合适的级别;跑不了浏览器就降级到 functional,并说明原因 —— ——

⚙️功能 · functional

"API 返回 200""CLI 输出版本号"这类——后端/命令层。最快、最稳。

$ vf run --pr owner/repo#42 \
    --level functional --checkout

🌐UI · ui

"点导出按钮能下载 CSV"这类——要真在浏览器里看。先录登录态再验。

$ playwright codegen \
    --save-storage=auth.json
$ vf run --pr owner/repo#42 --level ui \
    --base-url https://preview… --ui-auth auth.json

🧭端到端 · journey

"用户从登录到完成支付"这类跨多步、跨前后端的旅程。最慢但最接近真实。

$ vf run --pr owner/repo#42 \
    --level journey --checkout
不确定选哪个?用 --level auto:它看工单要求的是"接口行为"还是"界面交互"来决定。 环境跑不了浏览器(比如没装 Playwright)时,它会自动降级到 functional 并在报告里说明为什么,不会假装验过。
到底提没提效率

它替掉的是"人肉确认工单"这件事

❌ 没有它的时候

  • Reviewer 看完代码,还得回去翻工单:"它说要做的几条,都做了吗?"
  • 手动拉分支、跑一遍、点一遍 UI 确认
  • "看起来对"就合了,工单没真正核对
  • 每个 PR 重复一遍,经验留不下来

✅ 有它之后

  • 工单标准自动逐条核对,带证据贴回 PR
  • 真 checkout + 执行,不靠"应该没问题"
  • 失败可设成阻断合并,守住交付质量
  • 测试点进记忆,下次同类标准秒复用

VerifyFlow 的强项

  • 验的是"有没有兑现工单",不是"测试过没过"——这是 CI 不知道的事
  • 真执行 + 证据 + 记忆,比无状态 LLM judge 扎实得多
  • 能接合并门,能在流水线里自动闭环
  • 判错有正规纠错通道(反馈回路)

还要诚实承认的

  • 结果有随机性(同代码跑两次可能不同)——已立项修
  • "内部逻辑型"标准它会瞎编 shell 命令,不如直接跑单测准
  • 偏保守,常出一堆 not_evaluable,需要人收尾
  • 替代不了 CI 和代码审查,是补位不是顶替

注意:不是竞争关系。VerifyFlow 的 ui 级别本身就建在 Playwright 这类浏览器引擎之上;它做的是在上面加一层"绑工单 → 判交付 → 留证据 → 进合并门"。

案例 ①(功能):"确认 PR #66 把 README 工单的 12 条都做到了"

CodeRabbit / code-review skill

读 diff,给"这段写得好不好"的评论。但它不执行,也不知道工单要求那 12 条——没法回答"做到了没"。

verify skill

你得自己说"去核对这 12 条";它跑一下、对话里回你结论。跑完即忘,不贴回 PR、不进门、下次还得重说一遍。

VerifyFlow(本次真实)

自动从工单抽出 12 条 → 逐条跑探针 → 判定表贴回 PR #66 → AC-5 误判被 vf feedback 纠正 → 重验后 PR 评论自动刷新。全程留证据。

Playwright-MCP

这是浏览器任务,功能/文档型标准用不上它。

案例 ②(UI):"验证'点导出按钮能下载 CSV'"

Playwright-MCP / browser-use

能把浏览器开起来点按钮——而且很强。但它只是执行你下的指令,不知道这对应工单哪条标准,不给"交付判决",不沉淀证据/记忆,也不接合并门。

VerifyFlow --level ui

底层用同一套 Playwright 开真浏览器点按钮,但把这一步挂到工单那条标准上,产出 pass/fail + 截图证据 + 汇总判决,失败能阻断合并。

案例 ③(端到端):"验证'用户能注册 → 下单 → 看到确认页'" (示意场景 · journey 级别本次未实跑)

Playwright-MCP / browser-use

能把整条流程在浏览器里走通,执行力最强。但步骤要你编排,它不知道这对应工单哪条标准,不产出交付判决,不串起可追溯的证据链,也不接合并门。

手写 E2E 脚本(Cypress / Playwright)

确定、可复现、最可靠。但要人写、要维护,而且和工单脱节——覆盖什么由写脚本的人定,不是工单定。

VerifyFlow --level journey

把这条多步旅程挂到工单那条标准上:后端操作 + 浏览器操作串起来走,每步留证据,汇总成交付判决,失败能拦合并。

诚实短板

流程越复杂、越长,VerifyFlow 的稳健性不如手写脚本;它赢在"绑工单 + 判交付 + 进流水线",不在"执行得多稳"。

什么时候别用 VerifyFlow(用别的更合适)

👉 想查代码 bug / 安全 / 风格

用 CodeRabbit / Crosscheck / code-review skill。VerifyFlow 压根不审代码质量。

👉 只想快速确认"这改动有没有效"

用 verify skill。不用建工单、不用配 repo,一句话就跑,比 VerifyFlow 轻得多。

👉 要写复杂、可复现的 UI/E2E 脚本

直接用 Playwright(MCP)。它的浏览器自动化能力和确定性都强过 VerifyFlow 的 ui 级别。

👉 要快、要稳、要 required check

用 CI / 单元测试。确定、秒级、可信;VerifyFlow 慢且有随机性,是补位不是顶替。

平衡的一句话:VerifyFlow 唯一真正比别人强的地方,是"把交付对照 Linear 工单、真执行、留证据、能进合并门"这一整套——别的工具单点更强(审代码、控浏览器、快),但都没把这条链闭上。 它的代价也实在:慢、有随机性、偏保守、要好工单、还很早期。选它,是因为你要的是"工单到底兑现没有"这个特定问题,而不是因为它哪都强。
和 CI/CD 单元测试的区别

CI 绿 ≠ 交付对

最常被问的一个问题:"我都有 CI 单测了,为什么还要它?" —— 因为它俩回答的是两个不同的问题。

CI / 单元测试VerifyFlow
回答的问题"现有测试过不过?有没有弄坏已有行为?""这个 PR 有没有兑现工单要求的那几条?"
知道 Linear 工单吗✗ 完全不知道✓ 从 PR 反推工单,逐条对验收标准
"测什么"由谁决定写测试的开发者工单的验收标准
确定性 / 速度✓ 确定、秒级、可做 required check慢些;靠缓存才稳定(IN-801)
内部逻辑对不对✓ 写好的单测最可靠✗ 弱(会现编 shell 命令)
关系互补。CI 是地基;VerifyFlow 站在上面加"交付对没对工单"这一层——而且功能级别还会顺带跑你的测试套件当作其中一个探针。
关键例子:一个 PR 可以所有单测全绿(没弄坏任何旧东西),但工单要的 3 条里只做了 2 条—— CI 不会报,因为没人给"工单第 3 条"写过测试。VerifyFlow 会,因为它把那 3 条从工单里抽出来逐条验。
反过来也成立:"这段逻辑算得对不对"这种,写好的单元测试比 VerifyFlow 现编命令靠谱得多。 所以别拿它替 CI——让 CI 守"代码正确",让 VerifyFlow 守"交付了工单"。两条腿走路。
边界

它不做什么

  • 不审代码、不判可合并性 —— 那是 Crosscheck
  • 不拆工单、不派编码 agent —— 那是 Symphony / Jazzband
  • 不动钱、不发生产邮件、不做不可逆副作用
  • 不替代 CI
一句话总结:VerifyFlow 把"这个 PR 到底有没有兑现工单"这件原本靠人肉确认的事,变成了 可自动执行、带证据、可纠错、能进流水线的一环。它不完美(随机性、保守),但在 "Symphony 写 → Crosscheck 审 → VerifyFlow 验"这条链上,它补的是最后也最关键的一环:交付是否为真。
