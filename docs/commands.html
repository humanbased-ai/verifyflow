自动验证 PR 是否真正交付了 Linear ticket 里的需求，生成可追溯证据，贴回 GitHub PR comment。

git clone && npm install && npm link
环境准备（首次执行一次）

下面所有例子用的是 vf <command>。要让这条命令直接可用，先按 方式 A 把 vf 软链到全局 PATH；或者一直加 npm run vf -- 前缀（方式 B，不需要 link）。

$
git clone git@github.com:humanbased-ai/verifyflow.git && cd verifyflow # 拉仓库
$
npm install # prepare 钩子会自动跑 tsc，生成 dist/
$
# 方式 A · 推荐：把 vf 装到全局 PATH，下面例子原样可跑
$
npm link # 之后任意目录 `vf doctor` / `vf run` 都能用
$
# 方式 B · 不 link：把例子里的 `vf X` 改成 `npm run vf -- X`
$
npm run vf -- doctor # 等价于 `vf doctor`，只在 verifyflow 目录内可用

实战场景 —— 我该敲什么
🧑‍💻「我刚把 PR 推上去，想确认它是不是真的把 ticket 的需求做完了」

在目标 repo 目录里直接跑，零参数：从当前分支推断 PR，从 PR 正文里的 Linear 链接找 ticket，自动选验证级别。

$
cd ~/code/my-app && vf run
👀「我不在 repo 目录，只有一个 PR 号，想看看 VerifyFlow 打算怎么验，但先别真跑」

--dry-run 把解析出来的验收标准 + 计划中的探针打印出来，不 checkout、不执行、exit 0。

$
vf run --dry-run --pr humanbased-ai/verifyflow#54
🛡️「验证没过就别让这个 PR 合进来，并且把结论贴到 PR 上」

真执行（--checkout 拉 PR head），把报告贴成 PR comment，--policy merge_gate 让 needs_fix 时 exit 1 卡住合并。

$
vf run --pr owner/repo#42 --level auto --checkout --comment --policy merge_gate
🌐「这是个前端改动，要真的开浏览器点一遍，而且页面需要登录」

先用 Playwright 录一份登录态，再用 --level ui 驱动浏览器验证；--base-url 指向运行中的 app（省略则尝试从 PR 的 check 里找部署预览）。

$
playwright codegen --save-storage=auth.json https://app.local
$
vf run --pr owner/repo#42 --level ui --base-url https://app.local --ui-auth auth.json
🤷「这个 PR 压根没关联 Linear ticket，但我还是想验一下」

降级模式：拿 PR 自己的描述当验收标准，verdict 最高只到 manual_review_required。

$
vf run --pr owner/repo#42 --allow-no-ticket
🤖「我懒得每个 PR 手动跑，让它自己盯着 repo，过了就自动合」

独立 daemon：监听该 repo 里 Crosscheck 批准过的 PR，自动验证，accept 时 squash-merge。

$
vf watch --repo humanbased-ai/verifyflow --auto-merge --level auto
📊「想回顾一下这个 repo 最近一个月验证质量的趋势」

把累积的历次 run 聚合成质量指标，按 repo + 时间过滤，--trend 出趋势。

$
vf report --trend --repo owner/repo --since 2026-05-01
🔁「上次那个判决我觉得不对，改了判决逻辑想重判，但别再跑一遍探针」

用存储的证据重跑 verdict 引擎——不重新 checkout、不重新执行探针。runId 在每次 vf run 输出里。

$
vf replay run_2026-05-30_a1b2c3
🎼「我在 Symphony 编排里调 VerifyFlow，要一行机器可读的 JSON，绝不能卡流程」

orchestrator 专用：advisory-only 永不阻断，自动从 PR 解析 Linear issue，checkout + 执行 + 贴 comment，stdout 输出单行 JSON。

$
vf step --pr owner/repo#42 --crosscheck-verdict approved
🧪「我连账号都还没配好，就想先看看 VerifyFlow 跑出来长啥样」

离线 demo：不需要任何凭证，跑内置 fixtures 并生成一份可读报告，--open 直接打开。

$
vf demo --open

验证级别 —— --level
级别 做什么 速度 需要
functional 命令行探针 + 测试套件 快 checkout
ui Playwright 驱动浏览器验证 UI 中 Playwright + 运行中的 app
journey 多步骤端到端（backend + browser） 慢 checkout + Playwright
auto 根据 ticket 内容自动选最合适的级别 — —
合并门控 —— --policy
策略 行为
advisory 默认，只报告，永不阻断
merge_gate verdict = needs_fix 时 exit 1，阻断合并
strict manual_review_required / accept_with_risks 也阻断
自动化 daemon — vf watch
$
vf watch --repo owner/repo # 监听所有新 PR，自动验证
$
vf watch --repo owner/repo --auto-merge --level auto
$
vf watch --repo owner/repo --interval 120 # 轮询间隔 120s（默认 60）
目标 repo 配置 — vf init
$
vf init # 在目标 repo 目录下跑，自动检测 npm/uv/go/cargo/make
$
vf init --dir /path/to/repo
历史记录 & 回放
$
vf report # 汇总质量指标
$
vf report --trend --repo owner/repo --since 2026-01-01
$
vf show <runId> # 重新渲染某次报告（runId 在每次 vf run 输出里）
$
vf replay <runId> # 用存储证据重跑判决，不重新执行探针
$
vf signal <runId> # 查看改进建议信号
测试点内存 — vf memory
$
vf memory ls # 列出所有存储的测试点
$
vf memory show <key>
$
vf memory clear --repo owner/repo # 清某个 repo 的记忆
$
vf memory clear --yes # 清全部（跳过确认）
